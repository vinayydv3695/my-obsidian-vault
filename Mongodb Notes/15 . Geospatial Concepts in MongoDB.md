 


# # **1. Why Geospatial Data Matters**

MongoDB supports geospatial data for real-world applications such as:

- Food delivery
    
- Ride sharing
    
- Navigation & maps
    
- Geo-based filtering
    
- Distance calculations
    

It uses **WGS84** datum and **GeoJSON** geometry formats.

---

# ---------------------------------------------------------

# # **2. Earth Models — Geoid vs Ellipsoid**

### **Geoid**

- Real, irregular shape of Earth
    
- Based on gravity variations
    
- Mountains, oceans affect shape
    

### **Ellipsoid**

- Smooth, mathematical model of Earth
    
- Used in calculations
    
- Used in mapping systems
    

**MongoDB + GeoJSON use ellipsoid model (WGS84).**

---

# ---------------------------------------------------------

# # **3. What is a Datum?**

A **datum** defines the reference framework for latitude & longitude measurements.

Examples:

- **WGS84** — global standard (used by GPS, MongoDB, GeoJSON)
    
- NAD83
    
- GDA94
    

Datums define:

- Orientation of ellipsoid
    
- Zero point
    
- Shape used for mapping
    

MongoDB **always expects WGS84**.

---

# ---------------------------------------------------------

# # **4. Latitude & Longitude Basics**

### **Latitude (φ)**

- North/South from Equator
    
- Range: −90° to +90°
    

### **Longitude (λ)**

- East/West from Prime Meridian
    
- Range: −180° to +180°
    

Together → represent ANY location on Earth.

### **Flat vs Spherical Geometry**

- **2d**: flat plane → faster but less accurate
    
- **2dsphere**: spherical Earth → accurate real-world distance
    

MongoDB supports **both**, but 2dsphere is recommended.

---

# ---------------------------------------------------------

# # **5. GeoJSON Geometry Types**

MongoDB supports standard GeoJSON types:

### **Point**

A single location.

### **MultiPoint**

Multiple points.

### **LineString**

Series of points forming a line (roads, routes).

### **MultiLineString**

Multiple lines.

### **Polygon**

Closed boundary (city, lake).

### **MultiPolygon**

Multiple polygons.

### **GeometryCollection**

Collection of any geometry types.

---

# ---------------------------------------------------------

# # **6. GeoJSON Syntax Examples**

### **Point**

```json
{
  "type": "Point",
  "coordinates": [-20.22, 23.2]
}
```

### **LineString**

```json
{
  "type": "LineString",
  "coordinates": [
    [-20.22, 23.2],
    [-21.00, 22.2]
  ]
}
```

### **Polygon**

```json
{
  "type": "Polygon",
  "coordinates": [[
    [-20.22, 23.2],
    [-21.00, 22.2],
    [-20.00, 21.2],
    [-20.22, 23.2]
  ]]
}
```

# ---------------------------------------------------------

# # **7. MongoDB Geospatial Indexes**

MongoDB supports:

### **1. 2d index**

- Flat geometry
    
- Legacy use cases
    

### **2. 2dsphere index**

- Default for GeoJSON
    
- Uses spherical distances
    
- Works with GPS & real-world coordinates
    

Example:

```javascript
db.cities.createIndex({ location: "2dsphere" });
db.restaurants.createIndex({ location: "2dsphere" });
```

---

# ---------------------------------------------------------

# # **8. Example: Find Nearby Restaurants**

```javascript
const lucknow = {
  type: "Point",
  coordinates: [80.9462, 26.8467]
};

db.restaurants.find({
  location: {
    $near: {
      $geometry: lucknow,
      $maxDistance: 300000   // 300 km
    }
  }
});
```

- `$near` sorts by proximity
    
- `$maxDistance` filters by radius

# # **10. Key Takeaways**

- Geoid → real Earth; Ellipsoid → smoothed mathematical version
    
- WGS84 is universal (GPS, Google Maps, MongoDB)
    
- GeoJSON is the standard for storing geospatial data
    
- Use **2dsphere index** for accurate queries
    
- `$near`, `$geoWithin`, `$geoIntersects` for spatial queries
    
- You can visualize data with online tools before importing
