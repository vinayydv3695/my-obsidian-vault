# ## 1. What is a Graph Database?

A **Graph Database** is a NoSQL database used to store and query **nodes (entities)** and **relationships (edges)**.

### **Key Features**

- Stores highly connected data
    
- Schema-flexible
    
- Supports **property graph** & **RDF models**
    
- Optimized for fast relationship traversal
    

### **Examples**

- Neo4j
    
- Amazon Neptune
    
- ArangoDB
    
- Cosmos DB
    

---

# ## 2. Property Graph Model

A widely used graph model.

### **Elements**

- **Nodes:** Represent entities (Employee, Department)
    
- **Edges (Relationships):** Represent connections (WORKS_IN, LIKES)
    
- **Properties:** Key–value pairs stored inside nodes & edges
    
- **Labels:** Categorize nodes (e.g., Movie, User, Product)
    

### **Example**

Employee —[:WORKS_IN]→ Department

Nodes and relationships both can have properties.

---

# ## 3. RDF Model (Resource Description Framework)

A W3C standard for linked data.

### **Structure**

Data is represented as **Triples**:

- **Subject**
    
- **Predicate**
    
- **Object**
    

### **Example Triple**

Employee1 — WorksIn → HR

### **Use Cases**

- Semantic web
    
- Knowledge graphs
    
- Data exchange & reasoning
    

---

# ## 4. Graph vs Relational Databases

### **Graph Databases**

- Store nodes + edges directly
    
- No JOINs
    
- Flexible schema
    
- Very fast relationship traversal
    

### **Relational Databases**

- Store data in tables
    
- Relationships via foreign keys
    
- Schema updates require migrations
    
- Connected-data queries need multiple JOINs
    

### **Example**

Finding a mentor relationship:

- SQL → 3–5 JOINs
    
- Graph DB → simple traversal ("two hops")
    

---

# ## 5. Graph vs Other NoSQL Databases

|Database Type|Strength|Limitation|
|---|---|---|
|**Key-Value**|Extreme speed|No relationships|
|**Document**|Flexible, nested docs|Relationships weak|
|**Column**|Heavy analytics|Not for connected data|
|**Graph**|Relationship queries|Not ideal for flat data|

### **Graph DB Strength**

- Best for **highly connected** and **relationship-heavy** data
    

---

# ## 6. When to Use Graph Databases

### **Ideal For**

- Social networks
    
- Fraud detection
    
- Recommendation engines
    
- IT infrastructure
    
- Knowledge graphs
    
- Identity & access systems
    

### **Not Ideal For**

- Flat, simple data
    
- Heavy aggregations
    
- Large-scale numerical analytics
    

---

# ## 7. Graph Theory Basics

### **Key Concepts**

- **Directed edges:** A → B
    
- **Undirected edges:** A — B
    
- **Weighted edges:** With cost/distance
    
- **Cycles:** Paths that return to the starting node
    

### **Traversal Algorithms**

- **BFS (Breadth-First Search):** Layer-by-layer
    
- **DFS (Depth-First Search):** Deepest path first
    

---

# ## 8. From Requirements to Graph Model

### **Steps**

1. Identify **entities** (nodes)
    
2. Identify **relationships** (edges)
    
3. Assign **labels**
    
4. Add **properties** to nodes & relationships
    
5. Define directions & cardinalities
    
    - 1–1, 1–N, N–1, N–N
        

---

# ## 9. Key Terminologies (Neo4j Property Graph)

- **Node:** Entity (Person, City, Product)
    
- **Relationship:** Connects nodes
    
- **Label:** Categorizes nodes
    
- **Property:** Key–value data stored inside nodes/relationships
    

---

# ## 10. Cypher Query Language (Neo4j)

A **declarative** graph query language.

### **Example Query**

```
MATCH (e:Employee)-[:WORKS_IN]->(d:Department)
RETURN e.name, d.name
```

### **Important Clauses**

- CREATE
    
- MATCH
    
- OPTIONAL MATCH
    
- WHERE
    
- RETURN
    

---

# ## 11. Gremlin Traversal Language

An **imperative** graph traversal language.

### **Example**

```
g.V().hasLabel('Manager').out('manages').hasLabel('Employee').values('name')
```

### **Features**

- Step-by-step traversal
    
- Works across multiple graph systems
    
- More procedural than Cypher
